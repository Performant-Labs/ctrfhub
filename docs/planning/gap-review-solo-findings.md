## Resilience and operations gaps

- [x] **AI pipeline restart recovery for A2 / A3 is missing** (P0) — *resolved 2026-04-22: added DD-017 to `database-design.md` establishing `ai_pipeline_log` as the durable source of truth for pipeline scheduling (not a new `ai_jobs` table — reused the existing observability table by extending it). Added `worker_id`, `heartbeat_at`, `attempt` columns plus `UNIQUE(test_run_id, stage)` and `ON DELETE CASCADE` in §4.8. Reserve-execute-commit lifecycle: stage handlers `INSERT … ON CONFLICT DO NOTHING`, then atomically transition `pending → running` with heartbeat; worker crash is detected by stale `heartbeat_at` (> 2 min) and the row is reset to `pending` by (a) an on-boot scan and (b) a 60-second sweeper. Added an idempotency guard: stage checks its primary output column (e.g. `ai_root_causes_at`) before calling the LLM to avoid double-billing on recovery. Updated `ai-features.md` Implementation Architecture with a new "Durability and restart recovery (A1–A4)" subsection that points to DD-017 and summarises the lifecycle. DD-016's A1-specific startup query is kept alongside DD-017 — the two recoveries are idempotent with respect to each other.*

- [x] **SQLite + AI writer contention unsized** (P2) — *resolved 2026-04-22: added two new subsections to `deployment-architecture.md` under "SQLite deployment": (1) "SQLite writer contention with the AI pipeline" — enumerates the DD-017 write budget (reservation, 15s heartbeat, commit) alongside ingest and per-result A1 updates, notes WAL mode's one-writer-at-a-time rule, and identifies what breaks first (concurrent ingest bursts; A1 fan-out UPDATEs; UI writes colliding). (2) "When to graduate to PostgreSQL" — replaces the outdated "50 runs/day" hand-wave with a defensible table (≤ 200 runs/day, ≤ 3 peak concurrent ingests/60s, ≤ 5 concurrent users, ≤ 500 avg results/run, ≤ 5 in-flight AI runs) plus observable signals (p99 ingest latency > 2s, `ai_pipeline_log` `pending` rows with `attempt > 0`, SQLITE_BUSY log rate, UI write latency > 500ms). Explicitly framed as planning estimates, not measurements — ties back to gap-review item #17 (AI pipeline load-test scenario) as the mechanism to replace the estimates with real numbers. Stale "50 CI runs/day" line removed from the SQLite limitations bullet list and replaced with a pointer to the new threshold table.*

- [x] **Migration healthcheck race window** (P2) — *resolved 2026-04-22: canonical endpoint is `/health`, not `/healthz` as the checklist had it.
