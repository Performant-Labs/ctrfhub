## Verification performed (T1 hand-offs)

The audit-scope flagged two observations carried over from `audit-auth` for resolution here. Both are addressed explicitly.

- **(a) `src/app.ts:176` — `buildAuth(options.db)` as a DI seam.** **Conformant.** `buildAuth(dbPath?: string)` takes an optional SQLite-path override (`src/auth.ts:126`). When `DATABASE_URL` is set the dbPath is ignored and PG is used; otherwise SQLite is opened at `dbPath ?? SQLITE_PATH ?? ':memory:'`. The composition root forwarding `options.db` is the documented integration-test path (called out verbatim in `src/auth.ts` JSDoc lines 16–20) and is the right shape for the `AppOptions` seam: tests pass `db: ':memory:'` and both MikroORM and Better Auth get the matching in-memory database. The alternative (letting `buildAuth()` read env vars) would force tests to mutate `process.env` to inject a DB — strictly worse.
- **(b) `src/app.ts:561–582` — global auth preHandler's API-key branch.** **Drift (warn).** The branch ends in a definite allow/deny decision and **does not log the raw token value** — the SECURITY comment at lines 567–568 enforces this and the implementation honours it. The branch is therefore not a security `block`. However, T1's reading "logs presence-not-value" is **incorrect**: the branch logs *nothing* on invalid API key. Per `architecture.md §Code Conventions → Logging` line 859 ("the auth subsystem and ingest routes log decisions (e.g. invalid API key) without logging the token value"), the expected posture is to *log the decision* (presence + verdict) without the value. Filed as finding #6 below.

## Summary

The composition root holds the structural invariants well — `buildApp()` is the only wiring site, the layer chain is respected, the DI seam shape is sound, the EM-fork and global auth `onRequest` hooks register in the right relative order, the CSP directives match the architecture-doc table verbatim, and PR #71's `/assets/*` bypass + PR #77's inline-trivial-route clause for `/health` are correctly implemented. The drift is concentrated in three themes (see Themes below): **rate-limit configuration diverges from DD-012's canonical contract** (key and 429 response shape); **the AI-pipeline wiring is inlined at composition-root altitude** when it belongs behind a single `wireAiPipeline(...)` function in `src/services/ai/pipeline/`; **decorations and load-bearing ordering are undocumented** (no `FastifyInstance` augmentations for `orm`/`eventBus`/`artifactStorage`/`aiProvider`/`getBootState`/`setBootState`; LIFO `onClose` semantics relied on but uncommented; `onRequest` used where the architecture doc says "preHandler"). One observability gap (no log on invalid API key) and two minor consistency findings round out the list. No `block`-severity findings — the high-stakes auth preHandler is structurally sound.

## Themes

Findings group into three themes; each could be decomposed into a single story that fixes multiple findings at once.

**Theme T6-α: rate-limit + observability alignment with DD-012 / DD-029.** Findings #1 and #6. The composition root's rate-limit configuration is misaligned with the canonical Layer-2 table (wrong key, missing response-shape contract), and the canonical observability pattern (`event=*` Pino line on rate-limit / auth-failure decisions) is missing from the auth preHandler's API-key branch. Single story: bring the composition-root rate-limit + auth-failure logging into conformance with DD-012/DD-029.

**Theme T6-β: composition-root altitude — extract internal wiring, document load-bearing ordering.** Findings #2, #4, #7, and partially #11. The composition root has accreted lines that belong elsewhere: the AI pipeline subscriptions and sweeper (#2) belong behind `wireAiPipeline(...)` in `src/services/ai/pipeline/`; the LIFO `onClose` semantics (#4) need either a comment or restructuring; the EventBus runtime guard (#7) papers over test-double laxity that should be fixed at the test-double layer; and the boot-state 503 contract (#11) is structurally undeliverable because the server isn't listening when the boot states transition. Single story: tighten composition-root altitude — extract AI wiring, consolidate `onClose` into one explicit shutdown function, restructure boot to honour `/health` 503-during-sync, and tighten EventBus test seam.

**Theme T6-γ: small consistency cleanups — types, naming, doc/code alignment, redundant paths.** Findings #3, #5, #8, #9, #10. Five low-effort consistency fixes that together raise the composition root from "works" to "easy to maintain": FastifyInstance type augmentations (#3); rename `onRequest` → `preHandler` for the auth hook to match the architecture doc (#5); collapse the duplicated `/assets/` path-prefix in two branches (#8); cache the "users exist" boolean to remove the per-request COUNT (#9); decide on inline-vs-module home route (#10). Single story bundle: composition-root consistency cleanups.

## Out of scope but noticed

- **`recoverStalePipelineRows` runs as a fire-and-forget catch-and-log inside the composition root** (`src/app.ts:415-421`). The comment notes recovery failure is non-fatal because "the stuck-stage sweeper will catch them later." This is conformant with `architecture.md §Code Conventions → Error handling` ("Swallowed only when explicitly safe... the decision is commented at the call site"), but worth re-reviewing in a future audit of `src/services/ai/pipeline/` (T2 or a new AI-pipeline territory) — the boundary between "fatal at boot" and "non-fatal at boot" for AI-pipeline state is a policy decision currently buried in a comment.
- **`registerAuthRoutes` named-function variant vs. `ingestPlugin` default-export canonical shape.** Architecture.md line 745 explicitly tolerates this until AUTH-002 normalizes it; not a T6 finding (it's an auth-module-shape question), but worth confirming in the eventual AUTH-002 audit that the conversion did happen.
- **The four optional `AppOptions` DI seams (`db`, `artifactStorage`, `eventBus`, `aiProvider`) are the only DI surface.** No ambient module-level singletons have crept in (verified via grep on `services/event-bus.ts`, `services/ai/index.ts`, `lib/local-artifact-storage.ts`). The seam is healthy. Recommend a future T6-followup audit confirms this invariant holds after the dashboard / SSE / search modules land — those are the most likely sources of new cross-cutting dependencies.
- **`dist/assets/` vs. `src/assets/` invariant** (PR #71 — Operational Invariant): the `@fastify/static` root at line 235 is `path.join(__dirname, 'assets')`. In production, `__dirname` is `dist/`, so the served root is `dist/assets/` — conformant with PR #71's "production serves from `dist/assets/`" invariant. The builder-stage `cp -r src/assets/. dist/assets/` is owned by the Dockerfile, not by `app.ts`, so no T6 finding — but a Dockerfile audit (separate territory) would verify the bridge copy still exists.
- **`auth.ts`'s `getAuth()`/`_auth` dead singleton** is explicitly out of T6 scope per the audit-scope notes (line 58); `grep` confirmed `app.ts` does not import either symbol from `auth.ts`, so no transitive T6 finding is needed. PR #80 lands the deletion in T1's territory.
