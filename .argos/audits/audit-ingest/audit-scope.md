## Scope

**Paths to walk:**
- `src/modules/ingest/routes.ts` (310 LOC — the `POST /api/v1/projects/:slug/runs` route and the multipart-vs-JSON branching)
- `src/modules/ingest/schemas.ts` (306 LOC — Zod schemas: `CtrfReportSchema`, `IngestSuccessResponseSchema`, idempotency-key shape, multipart-field shape)
- `src/modules/ingest/service.ts` (405 LOC — `IngestService`: the persistence service, the 500-row chunked-insert pattern, idempotency lookups, transaction boundaries, `ReferenceOnlyError`)
- `src/modules/ingest/size-limit.ts` (44 LOC — `parseMaxJsonSize`, the JSON-payload size cap parser)

**Paths to ignore (within walk):**
- No test files live inside `src/modules/ingest/` — tests for the route live under `src/__tests__/integration/ingest.test.ts` and `src/__tests__/integration/ingest-artifacts.test.ts`. Tests are **out of scope** for this audit (Spec-enforcer / Test-writer domain).
- `src/lib/artifact-validation.ts`, `src/lib/magic-bytes.ts`, `src/lib/artifact-storage.ts`, `src/lib/local-artifact-storage.ts`, `src/lib/s3-artifact-storage.ts` are **out of scope** here — they belong to territory T4 (`audit-artifact-storage`). The ingest route *consumes* them; flag their consumption at the boundary if drift is visible, but do not walk into them. Specifically, the `validateMagicBytes`, `checkFileSizeLimit`, and `checkRunTotalLimit` calls in `routes.ts` are part of T4's contract surface — if their *usage* in ingest is wrong, that's a T2 finding; if they themselves are wrong, that's a T4 finding (do not file the latter here).

**Depth of recursion:** unlimited — the subtree is small (1,065 LOC total) and flat.

**Specific subsystems or layers in focus:**

- **The public `/api/ingest` contract** — `POST /api/v1/projects/:slug/runs` is one of three load-bearing public-API contracts in MVP (the others are `/api/v1/search` and `/api/artifact`). Every CI integration depends on its stability. Per `skills/ctrf-ingest-validation.md` and `product.md §Feature 1 Acceptance criteria` the contract is: `x-api-token: ctrf_*` header (not a session cookie); 201 on success with `{ runId }`; 422 on Zod-validation failure; 401 on missing/invalid token; 413 on artifact size cap; 429 on rate limit. Artifacts are co-uploaded in the same multipart request; there is no separate `/api/artifact` ingest endpoint.
- **Route → service → EntityManager layering** — the canonical example named in `architecture.md §Layering and Dependency Direction` line 673–678: "`src/modules/ingest/routes.ts` delegates all ingestion logic to … `IngestService` … its JSDoc explicitly states it 'never knows about Fastify request/reply shapes.'" Verify the route layer holds no business logic, the service layer holds no Fastify types, and the entity-manager boundary is respected.
- **Zod-schema-first validation** — per `skills/zod-schema-first.md` and `architecture.md §Code Conventions → Validation`: a single `CtrfReportSchema` in `schemas.ts` is the source of truth. Verify the route does not redo validation in handler logic; verify the service receives the parsed/typed shape, not the raw body.
- **Transaction boundaries and the 500-row chunked-insert pattern** — per `skills/ctrf-ingest-validation.md §Chunked Bulk Insert` and `~/Sites/ai_guidance/frameworks/mikro-orm/conventions.md §Chunked Bulk Insert (Event Loop Yielding)`: bulk test-result inserts use 500-row batches with `setImmediate` yields between batches. Verify the chunk size, the yield mechanism, and the transaction surrounding the chunked writes are dialect-portable (no Postgres-only constructs).
- **Idempotency-key handling** — per `product.md §Feature 1` line 102 and `database-design.md §DD-019`: re-posting the same payload within 24 hours with the same `Idempotency-Key` header returns `200 OK` with the original `{ runId }` and `X-Idempotent-Replay: true`. POSTs without the header always create a new run. Verify the lookup, the response headers, the 24-hour window, and the `?on_duplicate=replace` qualified-token branch.
- **Error → HTTP-status mapping** — per the contract: 401 for missing/invalid token, 413 for artifact size cap exceeded, 422 for Zod validation failure, 429 for rate limit, 500 for unexpected. Verify each error branch ends in a definite status code; verify `ReferenceOnlyError` (a service-layer error) maps to the right status; verify no catch-all `500` swallows errors that should be `4xx`.
- **Public API contract stability** — JSON response shapes, header names, query parameters, and accepted content-types are all part of the contract. Flag any drift from `product.md §Feature 1 Acceptance criteria` and `database-design.md §DD-019` even if it "works."
- **Event publication** — per `architecture.md §Module boundaries`: the ingest route publishes `run.ingested` on the EventBus after persistence. Verify the event is published exactly once per persisted run (not zero, not twice; not on the idempotent-replay branch unless the spec says so), and the payload shape matches the AI pipeline's subscriber expectations.

## Architectural concerns — the audit checklist

A walks the scope with these dimensions in mind. **Public API contract stability, layering, Zod-schema-first validation, and transaction boundaries are expanded** per the high-stakes nature of this territory. Cite the skill / planning section that defines the baseline pattern.

- **Layering — route → service → EntityManager (expanded).** `routes.ts` should hold: schema parsing, multipart-vs-JSON branching, header reads, status-code selection, calling into `IngestService`. `service.ts` should hold: persistence orchestration, transactions, the chunked-insert loop, idempotency lookups, event publication. The service must not import `fastify` or `FastifyRequest`/`FastifyReply` types. The route must not call `request.em.persist()` directly — it goes through the service. — `architecture.md §Layering and Dependency Direction` (line 673–678 specifically), `skills/fastify-route-convention.md` if it exists.
- **Zod-schema-first validation (expanded).** `CtrfReportSchema` and friends in `schemas.ts` are the single source of truth. The route's Fastify schema config must reference these — no ad-hoc `z.object({ … })` inline in a handler. The service receives the parsed type, not the raw body. — `skills/zod-schema-first.md`, `architecture.md §Code Conventions → Validation`.
- **Transaction boundaries (expanded).** The TestRun insert + N TestResult inserts must be in **one transaction** so a partial write doesn't leave a half-persisted run. The 500-row chunked-insert pattern should yield to the event loop *between* chunks (via `setImmediate`) but not split the transaction. Verify the transaction is opened in the service (not the route), and that idempotency-key insertion is part of the same transaction. — `skills/ctrf-ingest-validation.md §Chunked Bulk Insert`, `architecture.md §Code Conventions → Transaction boundaries`.
- **Public API contract stability (expanded).** Every header name, query param, status code, and JSON-response shape this route emits is part of the public contract. Flag drift from:
  - `product.md §Feature 1 Acceptance criteria` (status codes, idempotency replay semantics, the 24-hour window).
  - `product.md §Feature 4 Acceptance criteria` (artifact co-upload — no separate `/api/artifact` endpoint).
  - `database-design.md §DD-019` (idempotency-key table shape, replay rules, `on_duplicate` query param).
  - `database-design.md §DD-029` (canonical 429 response shape — `{"error":"rate_limited","code":"too_many_requests","retry_after_s":<int>}` — verify the route conforms; this lives in the composition root's global rate-limiter, but the route's per-route rate-limit config also needs to honour the shape).
- **Idempotency-key handling.** Lookup is on `(project_id, idempotency_key)` (the canonical compound key per DD-019). Verify the window is 24 hours (not 12, not 48), `X-Idempotent-Replay: true` is set on the replay response, and the `?on_duplicate=replace` branch requires the `ingest:replace` permission on the token (not implicit on any token).
- **Error handling and `ReferenceOnlyError`.** Service-layer custom errors (`ReferenceOnlyError`) must map to a specific HTTP status — not bubble as a 500. Verify the route's `catch` blocks distinguish: ZodError (422), `ReferenceOnlyError` (422 or 409 depending on what it represents — confirm against the service-layer JSDoc), unknown (500 with a logged correlation ID, not the raw error).
- **Multipart-vs-JSON branching.** The route accepts both content types. Verify both branches: (a) extract the CTRF JSON, (b) validate it with the same `CtrfReportSchema`, (c) for multipart, iterate file parts with `validateMagicBytes` + size checks **before** any bytes touch storage, (d) hand the parsed model + artifact streams to the service in one call. The multipart branch must not partially write artifacts to storage and then fail validation.
- **Per-route rate limiting.** `routes.ts` should configure a per-route rate limit consistent with `database-design.md §DD-012 Layer 2` (the ingest row of the table). Verify the keying (`req.user?.id ?? req.ip`-equivalent at the API-token level, per DD-012), the numeric limit, and the response shape (DD-029). Cross-reference against the composition root's global default (set in PR #85 to match DD-012/DD-029) — the per-route config either reuses the default or honours the same shape.
- **Event publication.** `run.ingested` is published after the transaction commits, exactly once per persisted run. The payload should be `RunIngestedPayload` (the constant in `src/services/event-bus.ts` / `RunEvents`). Verify the event is **not** published on an idempotent replay (the AI pipeline has already processed that run; re-publishing would duplicate categorization).
- **Naming, file structure, error handling, logging.** Standard dimensions. `ingest` follows the canonical module shape (`routes.ts` / `schemas.ts` / `service.ts` named files) — the architecture doc's `§Code Conventions → File organization` line 737 explicitly names this module as the canonical example. Verify any divergence from that shape.
- **Abstraction level.** Service methods should be at "one logical unit of work" altitude — `ingest(parsedReport, artifacts) → { runId, replayed }`-shaped, not a string-of-helpers chain. The route should be at "translate HTTP to a service call" altitude — not orchestrate the chunked-insert loop itself.
- **Dual-dialect correctness.** The chunked-insert pattern, idempotency-lookup query, and any raw SQL must work on both Postgres and SQLite (the dual MVP target). Flag any Postgres-only construct (e.g. `ON CONFLICT ... DO UPDATE`, `RETURNING`, `JSONB` operators) without a dialect equivalent. — `skills/mikroorm-dual-dialect.md`.
- **No tests in scope.** Do not file findings on `src/__tests__/integration/ingest*.test.ts`. If the tests miss coverage of a behaviour the audit identifies as load-bearing, note it under `Out of scope but noticed` so a follow-up audit / test-writer pass can address it.

## Acceptance criteria for `findings.md`

- Each finding has: `#`, `severity` (`block` | `warn`), `file:line`, `drift dimension`, `finding` (1–3 sentences), `suggested remediation`, `estimated story size` (XS <1 hr / S 1–4 hr / M half-day / L full day+).
- Findings prioritized by severity then by leverage (a single root cause that fans out to multiple files ranks higher than a one-off nit).
- A `Themes` section groups findings that share a root cause — these turn into single decomposed stories.
- No PASS/BLOCK verdict on the file — the prioritized list is the result.
- An `Out of scope but noticed` section captures anything outside the walk worth a future audit.
- A `Files examined` section lists the files A read in full.

- **Public API contract is high-stakes.** Any change to header names, status codes, JSON response shapes, query params, or accepted content-types is `block`-severity if it conflicts with `product.md` or `database-design.md`. The contract is what every CI integration consumes; getting it wrong silently breaks every downstream pipeline.
- **The "canonical module" status of `ingest`.** `architecture.md §File organization` line 740 explicitly names `ingestPlugin` (the default-export `FastifyPluginAsync` in `routes.ts`) as the canonical shape — the auth module's `registerAuthRoutes` variant is the *tolerated* exception. Do not file a finding that `ingest` uses the canonical shape; that's the spec. Do file a finding if any sub-component of `ingest` diverges from this canonical (e.g. a helper that should be a service method instead).
- **Two seams carry-over from prior audits.**
  - **(a)** PR #85 (audit-composition-root-S1) added a `keyGenerator: (req) => req.user?.id ?? req.ip` to the composition-root rate limiter and an `errorResponseBuilder` honoring DD-029. The *route-level* rate-limit config in `ingest/routes.ts` predates that PR. Verify the per-route config doesn't shadow the global config in a way that drops the DD-029 contract — if it overrides `errorResponseBuilder` without honouring DD-029, that's a finding here.
  - **(b)** `src/lib/event-bus.ts` (a likely-duplicate of `src/services/event-bus.ts`) was flagged in `audit-scoping`'s T4/T5 seam notes. The ingest route imports `EventBus` from `src/services/event-bus.ts` per the module convention — if it imports the other one, that's a finding here AND a stronger signal for the T4/T5 duplication.
- **`ReferenceOnlyError` is service-layer semantics.** Per the service JSDoc (verify): it represents a TestResult that has been referenced by another row (e.g. an artifact pointer or a comment) and can therefore not be deleted/replaced. In the *ingest* context it surfaces on the `?on_duplicate=replace` branch when the old run still has referenced results. Map it to the right HTTP status — `409 Conflict` is the most idiomatic; verify whatever the code does is consistent with the JSDoc and the public spec.
- **Dead-code state on main.** As of `d8c0334`, the audit-composition-root campaign is fully closed (PRs #85 + #86 + #87 merged). The composition root's global preHandler at `src/app.ts:708` is `preHandler` (renamed from `onRequest` in S3). The ingest route's `config: { skipAuth: false }` (or default) flow goes through that hook. If the ingest route registers its own `onRequest` or `preHandler` hooks that bypass auth, that's a `block` finding.
- **Out-of-scope for ingest specifically.** The `src/auth.ts` `buildAuth()` factory, the global auth preHandler in `src/app.ts`, the entity definitions under `src/entities/`, and the artifact-storage abstraction under `src/lib/` are all out of T2's territory. If ingest's *consumption* of any of those is wrong, that's a T2 finding; if any of them is wrong internally, that's a different audit's finding.
